What is Lightmin Engine Lightmin Engine (a combination of the words Light and Minimalistic) is an Engine aimed for creating Websites. It aims to provide a simple, fast, easy and powerful way to create dynamic websites, that is, websites that behave like applications! Dynamic websites are responsive websites, which often behave like applications, having a loading screen when the user accesses that website, and then the user no longer leaves that same page and only its content, as well as the interface, changes as the user navigates and uses the website, exactly how an application behaves.

The Lightmin Engine aims to be an alternative for people who do not want or cannot use much more complex Frameworks/Engines that require a greater learning curve, such as Flutter, React, Angular, among others.

The Engine is written in pure PHP, HTML, CSS and JavaScript! This means that to use this Engine and develop your website, all you will need is a server that runs PHP and a program to edit HTML, CSS, JavaScript and PHP codes! You won't need anything more than that and you'll be able to create your dynamic website in a matter of minutes after understanding the basics of this Engine!

Lightmin Engine Logo What is the dynamic behind Lightmin? Lightmin is an engine for creating dynamic websites based on Pieces, Pages and a Client. These are the three main pillars of Lightmin. The Client behaves like the base design of your dynamic website, being the base interface for your dynamic website, like mobile apps. Once the user enters your dynamic website, they will be entering the Client and will not leave there, unless they access another website or close the browser tab. The Pages work like standard HTML pages, however, they are rendered INSIDE the Client and whenever the user clicks on a link or button, and it is necessary to change to a new page, the user will not leave the Client, but instead, the current Page inside the Client will be unloaded and then the next Page will be loaded and then displayed inside the Client! You can watch the video below to understand better. In the example video below, I will show how MEGA dynamic website works. It wasn't made using Lightmin, but it works exactly like websites made with Lightmin! Using the MEGA website as an example... The Pieces act like Components in applications. Basically, Pieces are REUSABLE Components in which you use HTML, JavaScript and CSS to create them. Once you create a Piece, such as example, a Smart Text Field, you can render that Piece as many times as you want within Pages! And yes! If you decide to edit your Piece in the future, you will only need to edit the Piece file and then all existing Pieces on any Page will be updated too! The concept of Pieces makes it easier to maintain, organize and develop your websites!

Add the concepts of Client plus Pages plus Pieces plus HTML, CSS, PHP and the JavaScript that you already know and are used to, and then we have the Lightmin Engine! What do I need before start? The Lightmin Engine was developed to enable developers to create their dynamic websites (or web applications) in the most enjoyable, quick and easy way possible. It was developed so that you don't need to have compiler programs or anything similar on your PC, all you need is your standard code editor. But there are some requirements necessary for you to use Lightmin, some are optional. See below. Prior knowledge of HTML, CSS, JavaScript, JSON, PHP and how to create/host Websites. A server that runs PHP. Your Apache/Nginx server cannot hide the file extension of PHP/HTM/JS/CSS among others in the Browser URL. Everyone who will access your website will need Browsers that support JavaScript. An HTML, JavaScript, CSS and PHP code editor. Client: What is it and how does it work The Client is the client.php file on your Website created with the Lightmin Engine. It is the CENTRAL part of your entire Website, and for a user to access your Website, they must access the client.php page (file) in their Browser.

When the user accesses your Website Client, they will first see a loading screen that is responsible for loading the modular content of your entire Website. All of the Engine native code (Javascript and PHP) will already be completely loaded and included in the Client even before the loading screen starts operating. In the next topic, more will be said about the loading screen.

The Client is the ONLY page on which the user will access your Website, and once he enters your Website, he will no longer leave the Client page. As the objective of Lightmin Engine is to allow the creation of Websites that behave like applications, when accessing the Client, the user will see the loading screen and after that he will be able to use his Website and then, as he navigates, it will not load new pages or enter other pages as is done on a standard website. The user will only be in the Client, and as he navigates the Client content will change to different contents.

This is done in a very simple way. The Client has its layout like any Website, however, at some point in the Client, usually in the place where the page content would be displayed, such as text, news or something like that, there is an element called Window. The Windows belongs to the Lightmin Engine and are elements that allow you to display a page there, and behave in a similar way to the HTML iframe.

In addition to Windows, there is another very important element, which is Pages. The Pages belong to the Lightmin Engine and will be discussed in more detail later in the documentation, however, they function as "website parts" where on each Page you place content from your Website, as you would on a normal Website which has different pages.

The time the user accesses the Client, and the load ends, the home Page of the Website will be loaded and displayed within the main Window of the Client. As the user will browse the Website, only the content of Window are updated and exchanging the Pages in a very intuitive and fluid way. But not only that. As the user will Browse the pages, the URL in the Browser is dynamically changing to refer to the current Page that the user is seeing, and if he copies this URL and access it later or on another device, the exact same Page will be loaded! This also enables links to your website specific Pages to be copied and shared, as in any standard static website. Client: Loading and compilation system As stated in the previous topic, Client is the centerpiece of your Website created with Lightmin Engine. At some point, everything will connect with Client, everything will pass by the Client and everything will be part of the Client in any instance.

On the loading and compilation screen, all content of your website will be loaded, except the contents present in the media and pages folders. Then, in the loading and compilation process, the contents of the libraries, pieces, screens folders and the manifest.json and settings.php files will all be loaded INTO the Client at the time of loading.

This is precisely why the contents of your client.php, Screens, Pieces and your Pages will be affected by the CSS code you have in the libraries folder and that is also why you can access The Lightmin Engine JavaScript API from anywhere on your website, whether from client.php, Pieces or Pages. The JavaScript code you write and put in your libraries folder will be accessible from anywhere in your website as well, either from client.php, Pieces or Pages. The native code of Engine, metadata and similar things will always be loaded on the Client by default, even before the initial load of the Client begins, as they are necessary items for the full operation of Engine, and their internal tasks and features. (Does not contain all the details, but it is an approximate summary) Even if all the native code of Engine and other things related to Engine is always loaded on Client, even before the load screen begins, I STRONGLY RECOMMEND that you only run JavaScript code or use the Lightmin Engine JS API, only after the initial loading of the Client finish. This will ensure that you will not face problems or inconcases with your code, as loading can have performance and different variables while being underway. Therefore the safest is to wait for the Client load to be finished. As stated here, you can use the LE_OnLoadEngineFinished() callback to find out when the full Client loading has been finished. Click here and read the topic for more details about the LE_OnLoadEngineFinished() callback. Loading stages The loading and compilation that Engine makes in Client has a total of 10 stages. Each stage is responsible for doing a specific task to ensure the best organization and better operation of your website. Stage 1 At this stage, Engine removes some tags from the Client header. These are script tags generally included automatically on any AWS hosted sites. They are somewhat invasive tags that will not miss your website. Stage 2 At this stage Engine will load for Client all settings defined in settings.php. Even the settings you do not define in the file will still be loaded, but with your standard values defined by Engine. After loading the settings, they are all stored in cache for future Engine appointments or yours, made from JS API. Stage 3 At this stage, all previously loaded settings are applied to the Client. Stage 4 At this stage the manifest.json file is loaded so that Engine can know more details of your Website. Some information such as a base title of the Website and standard Page is stored in the cache while other information is used for the upcoming loading stages. It is thanks to the manifest.json loaded is that Engine can know exactly what are the Libraries, Pieces and Screens existing on your website and thus can load them to Client. Stage 5 At this stage, all CSS libraries stored in the libraries/css directory and informed in manifest.json are loaded and included in the Client header, having an effect throughout the client. For CSS libraries to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the PHP extension. The content of the file needs to obey a syntax and XML structure. All CSS code needs to be inside a <style> root tag. If the CSS code has a z-index attribute, it cannot be greater than 922000. The CSS code cannot contain the characters < and > outside of comments. The order in which each file is informed in manifests.json is completely respected in the loading process as well. Stage 6 At this stage, all JavaScript libraries within the libraries/js directory and informed in the manifest.json are loaded and included in the Client header, starting to apply with part of the Client JavaScript code, as each one is loaded. For JS libraries to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the PHP extension. The content of the file needs to obey a syntax and XML structure. All JS code needs to be inside a <script> root tag. The order in which each file is informed in manifests.json is completely respected in the loading process as well. Stage 7 At this stage, all Pieces inside the pieces folder and informed in manifest.json are loaded. The JavaScript and CSS code of each Piece is loaded to the client and are active on the Client, while the JSON and HTML code of each Piece is stored in cache to be used whenever each component is rendered in Pages that use them. For Pieces to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the HTM extension. The name of the Piece file can only contain letters. The content of the file needs to obey a syntax and XML structure. All Piece code needs to be inside a <piece> root tag. The for attribute of the piece root tag needs to have the value lightmin-engine. The name attribute of the piece root tag must have the value equal to the name of the Piece file, however, without the extension. The tag style (containing CSS), variables (containing JSON), code (containing HTML) and script (containing JavaScript) need to be present in the Piece file. The CSS code of Piece cannot use ID Selectors. The CSS code of Piece cannot use Tag Selectors. The CSS code of Piece cannot use Type Selectors. The CSS code of Piece cannot use Keyframes. The Class Selectors of the Piece CSS code must have a name following the format ".Piece_PieceFileNameWithoutExtension_TheClassName {}". If the CSS code of Piece has a z-index attribute, it cannot be greater than 922000. The JSON code of Piece needs to have a valid syntax. No one Piece HTML code tags can use the style attribute for inline styles. No one Piece HTML code tags can use the id attribute. The HTML code of Piece cannot make use of the html, head, body, style and script tags. All JavaScript code of Piece must be within a class. There can only be one class in the JavaScript code of Piece. The class name involving all the JavaScript code of Piece must have a name that follows the format "class Piece_PieceFileNameWithoutExtension {}". The CSS code and JSON code of the Piece cannot contain the characters < and > outside of comments. Stage 8 At this stage, all Screens inside the sceens folder and informed in manifest.json are loaded. The HTML code of each Screen is loaded to the Client, but they are disabled until the Screen is enabled using the Lightmin Engine JavaScript API. Anyway, the HTML code of each screen will already be part of the Client. For Screens to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the HTM extension. The name of the Piece file can only contain letters. The content of the file needs to obey a syntax and XML structure. All Screen code needs to be inside a <screen> root tag. The for attribute of the screen root tag needs to have the value lightmin-engine. The name attribute of the screen root tag must have the value equal to the name of the Screen file, however, without the extension. No one Screen HTML code tags can use the style attribute for inline styles. The HTML code of Screen cannot make use of the html, head, body, style and script tags. Stage 9 At this stage, Engine checks the code present in client.php you wrote. He will ensure that the rules below are respected... No HTML code tags in client.php can use the style attribute for inline styles. The script tag cannot be used within client.php. The style tag cannot be used within client.php. At this stage, Engine also does some additional jobs, such as preparing all window tags present within Client for use, and other things. Stage 10 At the final stage, Engine only finishes loading work, removes the loading screen, run callbacks to warn that the load has been completed and other things. Client: Inserting a Window to display Pages Once you have completed the layout of your Client, leaving the place ready to display content, it is time to insert a Window in your Client. As you may have read previously in this topic, Windows work as standard HTML iframes, and are where Pages will be loaded and unloaded, as the user navigates, just like how an application works. The Windows can only be inserted in Screens (which you will see later in the documentation) and in the Client as we will see now.

To insert a Window in your Client or Screen, simply use the HTML code below, placing it in the location you dedicated to displaying content in your Website layout... <le.window type="main" identifier="Example" scalingmode="parent-content"></le.window> When you insert the Window using this code, the Engine will prepare the Window on the loading screen and allow you to load Pages into it easily using the Lightmin Engine JS API. Now, let's understand what each attribute of the le.window tag means exactly... Attribute "type" The Windows can be divided into 2 types, which are main and normal. The difference between the two is that main is superior to normal. Both are capable of loading Pages normally and without any problems, however, the Window type main has special treatment from the Engine.

The Windows of type main automatically load the Page defined in defaultPage in manifest.json as soon as the initial loading of the Website finishes. Furthermore, the URI of the Page that is currently being displayed in a Window of type main is always displayed in the Browser URL field, and will always reflect the Page that is currently displayed within it. And finally, if a user accesses your Website from a link that refers to a Page, the main type Window will start loading the link Page as soon as the initial loading of your Website finished. The normal type Windows do not have this URL linking mechanism. If the user accesses your Website using a link to the Client that does not mention any Page, then the default page defined in defaultPage in manifest.json will be loaded in the Window of type main automatically after initial loading of the Website.

However, the main type Window can only be inserted once throughout the Website (and only on the Client), while the normal type can be inserted in each Screen and can be inserted in the Client too, together with the Window of type main. So, in total it is possible to have two Windows in the Client, and one Window per Screen. It is recommended that you define your Website default Page in the manifest.json file. Attribute "identifier" The identifier attribute only informs the ID of that Window. This ID will be used to refer to Window whenever you use the Lightmin Engine JS API to perform any action involving Windows. Attribute "scalingmode" The scalingmode attribute defines how the Window will expand horizontally and vertically according to the size of the currently loaded Page content. The value you will define in this attribute is made up of 2 subvalues separated by a -. The first value refers to the horizontal scale and the second value after - refers to the vertical scale.

The two possible subvalues are parent and content. The parent informs that the scale of the Window will reach a maximum of the size of the parent tag of the Window, while the content defines that the scale of the Window will be defined according to the size of the content that currently appears within the Window.

So, if you see a Window with the value below in the scalingmode attribute, means that the Window can scale vertically according to the size of the content, while it will remain horizontally at the width of the parent tag, which is the div. <div style="width: 800px;"> <le.window type="main" identifier="Example" scalingmode="parent-content"></le.window> </div> Continuing... So, now that you know exactly how to instantiate a Window in your Client, let's now see what the rules are for a Window to be considered valid by the Engine, when instantiated in the Client. There must be at least one Window of type main in the Client. There cannot be more than two Windows on the Client. The attributes type, identifier and scalingmod must be defined in any le.window tag. The value of the type attribute must be main or normal, and cannot be anything other than that. A Window cannot have an identifier that is already being used in another Window that is in the Client or Screen. There can only be ONE Window of type main and at MINIMUM ONE Window of type main in the Client. The values of the scalingmode attribute can only be parent or content and must follow the X-Y or horizontalValue-verticalValue format. For example, parent-content. As seen previously, none of your Website JavaScript code should be executed before the Engine finishes the initial loading of your Website. The same goes for Windows. The Windows are only ready after the Engine has loaded, so only try to load Pages after the initial Engine loading has finished. Pages: Creating Pages for your Website To create Pages for your Website made with Lightmin Engine, it's very simple. In the pages folder of your Website, you must create a file with PHP extension with whatever name you want, as if you were creating any page on a normal Website. Within the pages folder you can also create sub-directories to organize your pages as you prefer.

Once you have created your Page PHP file, the code inside the file should look like this... <html> <head> <!-- Page to Client Redirector --> <script type="text/javascript" src="https://example.com/engine/utilities/page-to-client-redirector.js"></script> <!-- Page Metadata --> <title>Page Title</title> <meta name="description" content="Page Description"/> <meta name="image" content="https://example.com/page-image.png"/> </head> <script> /* Page Script */ function LE_OnPageLoad(){ //on load this page... } </script> <body> <!-- Page HTML Content --> </body> </html> As you noticed, the structure of a Page is very simple and is very similar to the structure of a common HTML page, however, with some differences. The main objective of this structure is to allow full compatibility with search engines such as Google or Bing, so that a Page file closely resembles a standard HTML page and can be indexed by the engines. The Page file extension also needs to be PHP to allow you to run PHP scripts if you so desire. But now, let's break down each part of the Page to understand everything better. HTML tag The html tag is the root tag of a Page file precisely to make it look like a standard page, as explained previously in the paragraph above. HEAD tag The head tag is the header of your Page. Inside the header there can only be the following code below, and NOTHING MORE than that. <!-- Page to Client Redirector --> <script type="text/javascript" src="https://example.com/engine/utilities/page-to-client-redirector.js"></script> <!-- Page Metadata --> <title>Page Title</title> <meta name="description" content="Page Description"/> <meta name="image" content="https://example.com/page-image.png"/> The objective with this is to keep the head tag as small and minimalistic as possible. You do not need and SHOULD NOT place anything more than that within the header of your Pages. In addiction, you cannot insert style, link or script tags within the Pages header. Now, let's understand what each item in the header means...

Let's start with this part... <!-- Page to Client Redirector --> <script type="text/javascript" src="https://example.com/engine/utilities/page-to-client-redirector.js"></script> This is the ONLY allowed JavaScript code that can be placed within the head tag of your Pages. If a user accesses the PHP file of your Page directly, instead of accessing the client.php file, the user will only see your Page, that is, he will see a page without formatting, only with the content of text/HTML of your Page, as it will not be seeing the Page from client.php.

Knowing this, the Lightmin Engine has a JavaScript file called page-to-client-redirector.js which, when placed in the head tag of your Page, will cause a user that is accessing your Page file directly, he will be redirected to client.php, but the Page he had accessed will be loaded immediately within client.php. Then, the user will see the Page inside client.php as it should be, thanks to this script. This script can be found in the engine/utilities directory and is a script that is part of the Engine precisely so you don't have to worry about it.

The best way to add this script to your Page is by providing an absolute URL for the page-to-client-redirector.js script to the src attribute of the script tag in your Page header. So, if you have a website domain test.com where your Lightmin Website is at the root, you will need to provide the absolute URL being https://test.com/engine/utilities/page-to-client-redirector.js.

And now, as for this part... <!-- Page Metadata --> <title>Page Title</title> <meta name="description" content="Page Description"/> <meta name="image" content="https://example.com/page-image.png"/> The purpose of this part is to provide the metadata of your Page. This is where you define the title (required), description (optional) and image (optional) of your Page. All this metadata will appear in the Client and when links to your Website are shared by people. They can also be used by search engines and other things, so it's important that you provide this data whenever possible on your Pages.

These are the only things that should be present in the header of your Pages, nothing more. SCRIPT tag Inside the script tag of your Page is where you should place ALL the JavaScript code exclusive of your Page. It is not recommended to place large JS codes, and keep in mind that this code will be compiled to the Client as soon as your page is loaded, so do not name methods, classes and variables with names that are already used in JS codes present in libraries/js of your Website.

The window.onload and document.onload events will not work here. So if you want to run code right after your Page finishes loading and appears in the Client, just include a function called LE_OnPageLoad(). The method LE_OnPageLoad() must only be placed in your Page JavaScript code and only ONCE PER Page. BODY tag Inside the body tag is where you should place all the text content, HTML code or Pieces (which will be discussed later in the documentation) of your Page, just as you would do with a standard HTML page. The exception here is that you cannot use style and script tags here. WARNING: If you disobey any rules, write the wrong HTML/XML syntax or something like that, the Page will not be able to be loaded in the Client. Pages: Loading Pages inside Windows At any time you wish, you can load Pages within any Window on your Website. To do this, you just need to use Lightmin JavaScript API. To load a page, you just need to call the Windows.LoadPage() method passing as a parameter the identifier of the Window in which you want to load the Page and the URI for the Page file to be loaded. Windows.LoadPage("ClientMainWindow", "example-page.php", null); For it to work, the Page file needs to be within the pages directory of your Website. Furthermore, the second parameter of the Windows.LoadPage() method takes into account that the pages directory is the root, so you don't need to enter a URI like pages/example.php, just enter example.php.

Whenever you load a new Page you should always only use the Lightmin API for this. The use of the HTML a tag like <a href="example.com"></a> is unnecessary on Websites made with Lightmin, unless you want to send the user out of the Client, or to another page of course!

Now, if you want to receive a Callback to know whether the Page loading was successful or not, just use the Windows.LoadPage() method passing a function in the parameters, as below. As you saw previously, Pages are loaded within Windows. This way, if the user copies the URL from the Browser and shares it, the person who accesses it will see the same Page as the user who shared the URL. This will keep your website working correctly. Pieces: What is it and how does it work The Pieces are equivalent to the concept of "Components" and "UserControls" in frameworks like React and WPF. They are a set of HTML, CSS and JavaScript codes that you can use and reuse, instantiating them as many times as you want within Pages. Whenever you instantiate a Piece, you can instantiate it with different data, as needed. Basically, Pieces allow you to write CSS, HTML and JavaScript code once, and then reuse that code as many times as you want.

You can, for example, create a Piece that is a Text Field with the appearance and behavior you want. To do this, you will need to create a Piece with the HTML, CSS and JavaScript code once. Once this is done, you will only need to instantiate the Text Field within your Page as many times as you want! In the future, if you need to edit the Text Field code, you only need to edit the code in the Piece file and that's it! Pieces also facilitate repetitive code management, maintenance and standardization of your Website components. The Pieces can only be instantiated inside Pages files using HTML code or the Lightmin Engine JS API at Runtime. Pieces: Creating a Piece To create a Piece for your Website, it's very simple. In the pieces folder on your Website, you must create a file with the HTM extension and a name that contains only letters. I don't recommend placing subdirectories inside the pieces folder.

Once you have created the HTM file of Piece, the code inside the file should look like this... <?xml version="1.0" ?> <piece for="lightmin-engine" name="Example"> <documentation type="text/md"> <!-- Piece Mini-Documentation --> </documentation> <style type="text/css"> /* Piece CSS Code */ .Piece_Example_SomeCssClass { background-color: #FF0000; } </style> <variables type="text/json"> { "declaredVar1":"defaultValueForThis", "declaredVar2":"defaultValueForThis" } </variables> <code type="text/html"> <!-- Piece HTML Code --> <div class="Piece_Example_SomeCssClass" test="__declaredVar1__">__declaredVar2__</div> </code> <script type="text/javascript"> class Piece_Example { //Piece JS Code... /* * ... the code ... */ //Engine callbacks... static OnLoad() { } static OnInstantiate(inWindowId, pieceInstanceId) { } static OnDestroy(inWindowId, pieceInstanceId) { } } </script> </piece> As you may have noticed, the structure of a Piece file follows the XML structure. Each part of the Piece is contained within key tags. Now, let's go through each part of the Piece to understand what each part means. Your Piece file must comply with XML syntax, otherwise it will not be loaded by the Engine. PIECE tag The piece tag is the root of a Piece file. There are two attributes in this tag which are for and name. The for attribute must contain the value lightmin-engine and the name attribute must contain exactly the same name that you gave to the Piece file, however, without the extension. DOCUMENTATION tag Inside the documentation tags is where you can insert text describing what your Piece does, the features, variables, etc. So that you or someone else can consult the Piece file and understand everything it can to do. This text will never be loaded into the Client and it is recommended that it be placed in commented form, for better highlighting. STYLE tag Inside the style tags is where you should place the CSS code for your Piece. You should not place the CSS code of your Pieces in the libraries directory. The CSS code here can only use Class Selectors, and they must be in the format .Piece_PieceFileNameWithoutExtension_TheClassName. The CSS code for your Piece is only inserted into the Client once. You also cannot use the characters < and > in the CSS code. VARIABLES tag Inside the variables tags is where you should place the JSON code of your Piece. The JSON code of a Piece is used to declare variables and their default values. Variables can be placed in the HTML code of the Piece and when the Piece is instantiated, you can assign different values to such variables, and in this way, the Piece will be instantiated using different values. This is how you can instantiate the same Piece several times, but with different content and data. The content inside the variables tags must respect JSON syntax, otherwise the Piece will not be loaded. You also cannot use the characters < and > in the JSON code of variables. CODE tag Inside the code tags is where you should place the HTML code of your Piece. This is the HTML code that will be added to the Client whenever your Piece is instantiated within a Page. Within code tags you cannot use html, head, body, style or script tags. Additionally, you cannot use the id and style attributes in any tags within the code tags. SCRIPT tag Inside the script tags is where you should place the JavaScript code of your Piece. This is where you must place the JavaScript code that will make your Piece work, if necessary, of course. You can also execute code whenever your Piece is loaded by the engine, registering function OnLoad(), you can also execute code whenever your Piece is instantiated somewhere, registering function OnInstantiate(inWindowId, pieceInstanceId). Finally, you can register code to run whenever your Piece is destroyed by registering the OnDestroy(inWindowId, pieceInstanceId) function. Instantiated Pieces are destroyed whenever the Page they are on is unloaded to load the next Page in a Window. Additionally, Pieces can also be destroyed if you request it using the Lightmin Engine JS API. The JavaScript code for your Piece is only inserted into the Client once. All JavaScript code for your Piece must be inside a class that has a name in the format class Piece_PieceFileNameWithoutExtension. Furthermore, there can only be one class in the JavaScript code of your Piece.

Within the class, within the JavaScript code of your Piece, all functions and variables need to be static in order for them to work. Only variables within functions can be local variables. The OnDestroy(inWindowId, pieceInstanceId) callback is executed right after your Piece instance is destroyed and disappears from the interface. Even though the callback informs the Window and the ID of the instantiated Piece that was destroyed, this information is only for control and to better know where and which instance of the Piece was destroyed. Therefore, avoid executing Asynchronous codes and codes that try to Reference the Piece that was destroyed. Since your Piece JavaScript code is only inserted once in the Client, this means that ALL instances of your Piece have access to the same JavaScript code, i.e., your Piece JavaScript code is not separated by instances where you have a copy of the JavaScript code for each instance. Because of this, your JavaScript code must rely on the Instance ID, Lightmin's JS API for interacting with Pieces, and so on to interact with your Piece instances. If you need to store primitive values (Bool, Float, Int, and String) separately for each instance of your Piece, you can use Lightmin Individual Data JS API for this. The Individual Data API basically consists of the Pieces.SetInIndividualDataOfPiece() and Pieces.GetOfIndividualDataOfPiece() methods. This API provides an easy and straightforward way to store simple data for each instance of your Piece. The downside is that the Individual Data API is only capable of handling primitive values and can have a noticeable performance hit if called multiple times per second.

If you need a more performant alternative that allows you to store individual JS values (like objects, arrays, and all the rest) in addition to primitive values for your Piece instances, a good alternative is the Individual Storage API. This is a slightly more complex API, and it should be explained more in the "Lightmin JS API" section of this documentation. See the Lightmin API reference section of this documentation for more details on API Individual Data and API Individual Storage. Continuing... When using Pieces, you need to keep in mind that the CSS and JS code for the Piece will only be added to the Client once. The HTML code will be added within the Page WHENEVER you instantiate the Piece in the desired location of Page. In the next topics of this documentation, you will learn how to instantiate Pieces in desired locations, within Pages, using HTML code or the Lightmin Engine JS API. This is a rule that I strongly recommend you follow while developing websites with Lightmin: All CSS and JavaScript code of your ENTIRE website MUST be inside your website libraries folder. JavaScript files should look like this, and must have the PHP extension... <?php /* This is a example JavaScript file for the Lightmin website */ ?> <script> function exampleFunction() { } </script> The events window.onload and document.onload will not work for your JavaScript libraries because they are loaded at runtime on the loading screen, so if you need to run some code right after your library script is loaded by the Engine, just include a method called LE_OnLoad_YourJsLibFileName() in your script. So, if you have a JavaScript library called MyJsLibrary.php, just include the function LE_OnLoad_MyJsLibrary() method in your script and it will be executed as soon as your script is loaded by the Engine. If you want to run code once the Engine has COMPLETELY finished initial loading the Website, you can add function LE_OnLoadEngineFinished() to one of your library JavaScript files. But add this function ONLY ONCE throughout your website. Only ONE of this function can exist PER Website, to avoid conflicts when receiving the callback. CSS files should look like this, and must have the PHP extension too... <?php /* This is a example CSS file for the Lightmin website */ ?> <style> .exampleCssClass { } </style> These files must have the PHP extension as this will allow code editors (like VS Code) to provide auto-completion while you write your CSS and JavaScript code, as well as allowing PHP code to be merged with your CSS and JavaScript code, if you so desire.

Either way, every file that contains CSS should go in directory libraries/css on your Website, while every file that contains JavaScript code should go in directory libraries/js on your Website. And of course, both type of files must comply with the examples mentioned above, otherwise the Engine will not be able to compile them at the time of initial loading, which could result in problems running your Website. Manifest File As you already know, websites created with Lightmin Engine can have multiple Pieces, Screens and Libraries. All of these items help to create your website and make your work easier when developing it. However, upon initial loading of your website, the Engine loads all these items and then compiles them in the Client, at runtime. Once compiled, they will be available for use across your website. However, for the Engine to know that these items exist, you need to inform it. And that's where the manifest.json file comes in.

The manifest.json file is a JSON file and contains some arrays in which you must enter the path to the Libraries (CSS and JavaScript), Pieces, Screens files and provides basic data about your website. A manifest.json file should look like this... { "baseTitle": "Example Website", "defaultPage": "pages/home.php", "cssLibraries": [ "libraries/css/example.php" ], "javascriptLibraries": [ "libraries/js/example.php" ], "websitePieces": [ "pieces/TextField.htm", "pieces/ImageDisplay.htm" ], "websiteScreens": [ "screens/Preferences.htm" ] } Now, let's see what each variable in manifest.json means. baseTitle: Defines the base title of your Website. This will be the text that comes before the title of the Page currently loaded in the main Window. defaultPage: Defines the default Page of your Website. It is the Page that will be loaded by default, when none is specified in the URL. cssLibraries: In this variable you must enter the path to each CSS file within the libraries folder on your Website. javascriptLibraries: In this variable you must enter the path to each JavaScript file within the libraries folder on your Website. websitePieces: In this variable you must enter the path to each Piece file within the pieces folder on your Website. websiteScreens: In this variable you must enter the path to each Screens file within the screens folder on your Website. Remember that you must provide the paths to the files, taking into account that the root folder is the same folder where file client.php is located. Be careful to not save file manifest.json containing syntax errors. If the file does not comply with the correct JSON syntax, the Lightmin Engine will not be able to load it correctly and your Website may be interrupted. Also, never remove any existing variables (array) from file manifest.json, even if the array is empty. Pieces.SetInIndividualDataOfPiece("ClientMainWindow", "instanceId", "CarBrand", "Honda"); Pieces.SetInIndividualDataOfPiece("ClientMainWindow", "instanceId", "CarYear", 2021); If you provide a new Value for a Key that has already received a Value before, then that Key will be updated to contain only the new Value provided.

To read the Value contained in a Key, you will only need to inform the Instance ID of a Piece, the identifier of the Window that contains it, and the Key that contains the Value to be read, as per the example below... var carBrand = Pieces.GetOfIndividualDataOfPiece("ClientMainWindow", "instanceId", "CarBrand"); var carYear = Pieces.GetOfIndividualDataOfPiece("ClientMainWindow", "instanceId", "CarYear"); That's all you need to know about the Individual Data API. Lightmin JS API: Pieces Individual Storage API The Individual Storage API provides a slightly more complex way to store data in Piece Instances individually, that is, each Piece instance can contain different data stored in it.

Since all instances of a Piece share the same Piece JavaScript code, you may want to store data for each Piece individually, but since Piece methods and variables need to be static, this is not possible. That's why this API can help.

This API provides a somewhat complex way to store data individually in Piece Instances. Although it is somewhat complex, this API can store any type of data (float, int, string, bool, object, reference, array, etc) while maintaining good performance. The downside is that this API is not "just use", it also requires some management on your part when developing your Piece that will use it.

If your Piece will use this API, then your Piece needs to notify the Engine whenever a new Instance of your Piece is created. This way, the Engine can allocate data storage space for each instance of your Piece. You can notify the Engine about created instances, using the OnInstantiate() callback that can be inserted in the JS code of your Piece, in addition to the Pieces.IndividualStorage.AllocateForPieceInstance() method, as shown below... class Piece_Example { static OnInstantiate(inWindowId, pieceInstanceId) { Pieces.IndividualStorage.AllocateForPieceInstance(inWindowId, pieceInstanceId); } } If no space is allocated for instances in your Piece, then you will not be able to access Individual Storage on your Piece.
In addition to informing whenever an instance of your Piece is created, your Piece also needs to inform the Engine whenever an instance of your Piece is destroyed, that is, when it is destroyed by the JS API or when the Page it is on is unloaded. This way, the Engine can deallocate the space that was previously reserved for your Piece instance. This notification can be done using the OnDestroy() callback that can be inserted into your Piece JS code, in addition to the Pieces.IndividualStorage.DisposeOfPieceInstance() method, as shown below... class Piece_Example { static OnDestroy(inWindowId, pieceInstanceId) { Pieces.IndividualStorage.DisposeOfPieceInstance(inWindowId, pieceInstanceId); } } If you do not notify the Engine when instances of your Piece are destroyed, the Engine will not deallocate the previously reserved spaces. Eventually, everything will pile up and may cause performance issues and other errors. When an instance of your Piece is destroyed, and your Piece notifies the Engine, the Engine will deallocate the storage space previously reserved for that instance of your Piece, meaning that the data that was stored there will also be discarded.
If you want to check if an instance of your Piece has space allocated for it in the Engine Individual Storage, you can use the Pieces.IndividualStorage.isSpaceAllocatedForPieceInstance() method, it will return true if there is a space allocated for the Piece instance, as shown in the example below... var isSpaceAllocated = Pieces.IndividualStorage.isSpaceAllocatedForPieceInstance("ClientMainWindow", "pieceInstanceId");
Now that you know the basics needed to use the Individual Storage API, let's move on to the interesting stuff. To save data into a Piece instance, you will only need to provide the Piece Instance ID, Window identifier, a Key (which serves as the name of a Value Slot) and the Value to be saved, as shown in the example below... Pieces.IndividualStorage.StoreInPieceInstance("ClientMainWindow", "instanceId", "CarBrand", "Ford"); Pieces.IndividualStorage.StoreInPieceInstance("ClientMainWindow", "instanceId", "CarYear", 2015); Pieces.IndividualStorage.StoreInPieceInstance("ClientMainWindow", "instanceId", "CarColors", ["Black", "Blue", "Orange"]); If you provide a new Value for a Key that has already received a Value before, then that Key will be updated to contain only the new Value provided.

To read the Value contained in a Key, you will only need to inform the Instance ID of a Piece, the identifier of the Window that contains it, and the Key that contains the Value to be read, as per the example below... var carBrand = Pieces.IndividualStorage.RetrieveOfPieceInstance("ClientMainWindow", "instanceId", "CarBrand"); var carYear = Pieces.IndividualStorage.RetrieveOfPieceInstance("ClientMainWindow", "instanceId", "CarYear"); var carColors = Pieces.IndividualStorage.RetrieveOfPieceInstance("ClientMainWindow", "instanceId", "CarColors");
If you want to view all the data currently stored in the Individual Storage of an instance of your Piece, you can use the Pieces.IndividualStorage.DebugPieceInstance() method, informing the instance ID of your Piece and the identifier of the Window in which it is, as below. This will display the data in the Browser Console. Pieces.IndividualStorage.DebugPieceInstance("ClientMainWindow", "instanceId"); But if you want to view ALL data currently stored in Individual Storage, from ALL instances of ALL Pieces of your Website, you can use the Pieces.IndividualStorage.DebugAllPiecesInstances() method, which will also produce a result in your Browser Console. Pieces.IndividualStorage.DebugAllPiecesInstances(); That's all you need to know about the Individual Storage API.
