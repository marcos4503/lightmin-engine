What is Lightmin Engine Lightmin Engine (a combination of the words Light and Minimalistic) is an Engine aimed for creating Websites. It aims to provide a simple, fast, easy and powerful way to create dynamic websites, that is, websites that behave like applications! Dynamic websites are responsive websites, which often behave like applications, having a loading screen when the user accesses that website, and then the user no longer leaves that same page and only its content, as well as the interface, changes as the user navigates and uses the website, exactly how an application behaves.

The Lightmin Engine aims to be an alternative for people who do not want or cannot use much more complex Frameworks/Engines that require a greater learning curve, such as Flutter, React, Angular, among others.

The Engine is written in pure PHP, HTML, CSS and JavaScript! This means that to use this Engine and develop your website, all you will need is a server that runs PHP and a program to edit HTML, CSS, JavaScript and PHP codes! You won't need anything more than that and you'll be able to create your dynamic website in a matter of minutes after understanding the basics of this Engine!

Lightmin Engine Logo What is the dynamic behind Lightmin? Lightmin is an engine for creating dynamic websites based on Pieces, Pages and a Client. These are the three main pillars of Lightmin. The Client behaves like the base design of your dynamic website, being the base interface for your dynamic website, like mobile apps. Once the user enters your dynamic website, they will be entering the Client and will not leave there, unless they access another website or close the browser tab. The Pages work like standard HTML pages, however, they are rendered INSIDE the Client and whenever the user clicks on a link or button, and it is necessary to change to a new page, the user will not leave the Client, but instead, the current Page inside the Client will be unloaded and then the next Page will be loaded and then displayed inside the Client! You can watch the video below to understand better. In the example video below, I will show how MEGA dynamic website works. It wasn't made using Lightmin, but it works exactly like websites made with Lightmin! Using the MEGA website as an example... The Pieces act like Components in applications. Basically, Pieces are REUSABLE Components in which you use HTML, JavaScript and CSS to create them. Once you create a Piece, such as example, a Smart Text Field, you can render that Piece as many times as you want within Pages! And yes! If you decide to edit your Piece in the future, you will only need to edit the Piece file and then all existing Pieces on any Page will be updated too! The concept of Pieces makes it easier to maintain, organize and develop your websites!

Add the concepts of Client plus Pages plus Pieces plus HTML, CSS, PHP and the JavaScript that you already know and are used to, and then we have the Lightmin Engine! What do I need before start? The Lightmin Engine was developed to enable developers to create their dynamic websites (or web applications) in the most enjoyable, quick and easy way possible. It was developed so that you don't need to have compiler programs or anything similar on your PC, all you need is your standard code editor. But there are some requirements necessary for you to use Lightmin, some are optional. See below. Prior knowledge of HTML, CSS, JavaScript, JSON, PHP and how to create/host Websites. A server that runs PHP. Everyone who will access your website will need Browsers that support JavaScript. An HTML, JavaScript, CSS and PHP code editor. (I highly recommend "VS Code" because of its level of customization and extensions) Creating your first application with Lightmin! PLACE HOLDER!!! Now you know the basics! Client: What is it and how does it work The Client is the client.php file on your Website created with the Lightmin Engine. It is the CENTRAL part of your entire Website, and for a user to access your Website, they must access the client.php page (file) in their Browser.

When the user accesses your Website Client, they will first see a loading screen that is responsible for loading the modular content of your entire Website. All of the Engine native code (Javascript and PHP) will already be completely loaded and included in the Client even before the loading screen starts operating. In the next topic, more will be said about the loading screen.

The Client is the ONLY page on which the user will access your Website, and once he enters your Website, he will no longer leave the Client page. As the objective of Lightmin Engine is to allow the creation of Websites that behave like applications, when accessing the Client, the user will see the loading screen and after that he will be able to use his Website and then, as he navigates, it will not load new pages or enter other pages as is done on a standard website. The user will only be in the Client, and as he navigates the Client content will change to different contents.

This is done in a very simple way. The Client has its layout like any Website, however, at some point in the Client, usually in the place where the page content would be displayed, such as text, news or something like that, there is an element called Window. The Windows belongs to the Lightmin Engine and are elements that allow you to display a page there, and behave in a similar way to the HTML iframe.

In addition to Windows, there is another very important element, which is Pages. The Pages belong to the Lightmin Engine and will be discussed in more detail later in the documentation, however, they function as "website pieces" where on each Page you place content from your Website, as you would on a normal Website which has different pages.

The time the user accesses the Client, and the load ends, the home Page of the Website will be loaded and displayed within the Window of the Client. As the user will browse the Website, only the content of Window are updated and exchanging the Pages in a very intuitive and fluid way. But not only that. As the user will Browse the pages, the URL in the Browser is dynamically changing to refer to the current Page that the user is seeing, and if he copies this URL and access it later or on another device, the exact same Page will be loaded! This also enables links to your website specific Pages to be copied and shared, as in any standard website. Client: Loading and compilation system As stated in the previous topic, Client is the centerpiece of your Website created with Lightmin Engine. At some point, everything will connect with Client, everything will pass by the Client and everything will be part of the Client in any instance.

On the loading and compilation screen, all content of your website will be loaded, except the contents present in the media and pages folders. Then, in the loading and compilation process, the contents of the libraries, pieces, screens folders and the manifest.json and settings.php files will all be loaded INTO the Client at the time of loading.

This is precisely why the contents of your client.php, Screens, Pieces and your Pages will be affected by the CSS code you have in the libraries folder and that is also why you can access The Lightmin Engine JavaScript API from anywhere on your website, whether from client.php, Pieces or Pages. The JavaScript code you write and put in your libraries folder will be accessible from anywhere in your website as well, either from client.php, Pieces or Pages. The native code of Engine, metadata and other things will always be loaded on the Client by default, even before the initial load of the Client begins, as they are necessary items for the full operation of Engine, and their internal tasks and features. Visual representation of the process being done.
(Does not contain all the details, but it is an approximate summary) Even if all the native code of Engine and other things related to Engine is always loaded on Client, even before the load screen begins, I STRONGLY RECOMMEND that you only run JavaScript code or use the Lightmin Engine JS API, only after the initial loading of the Client finish. This will ensure that you will not face problems or inconcases with your code, as loading can have performance and different variables while being underway. Therefore the safest is to wait for the Client load to be finished. As stated here, you can use the LE_OnLoadEngineFinished() callback to find out when the full Client loading has been finished. Click here and read the topic for more details about the LE_OnLoadEngineFinished() callback. Loading stages The loading and compilation that Engine makes in Client has a total of 10 stages. Each stage is responsible for doing a specific task to ensure the best organization and better operation of your website. Stage 1 At this stage, Engine removes some tags from the Client header. These are script tags generally included automatically on any AWS hosted sites. They are somewhat invasive tags that will not miss your website. Stage 2 At this stage Engine will load for Client all settings defined in settings.php. Even the settings you do not define in the file will still be loaded, but with your standard values defined by Engine. After loading the settings, they are all stored in cache for future Engine appointments or yours, made from JS API. Stage 3 At this stage, all previously loaded settings are applied to the Client. Stage 4 At this stage the manifest.json file is loaded so that Engine can know more details of your Website. Some information such as a base title of the Website and standard Page is stored in the cache while other information is used for the upcoming loading stages. It is thanks to the manifest.json loaded is that Engine can know exactly what are the Libraries, Pieces and Screens existing on your website and thus can load them to Client. Stage 5 At this stage, all CSS libraries stored in the libraries/css directory and informed in manifest.json are loaded and included in the Client header, having an effect throughout the client. For CSS libraries to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the PHP extension. The content of the file needs to obey a syntax and XML structure. All CSS code needs to be inside a <style> root tag. If the CSS code has a z-index attribute, it cannot be greater than 922000. The CSS code cannot contain the characters < and > outside of comments. The order in which each file is informed in manifests.json is completely respected in the loading process as well. Stage 6 At this stage, all JavaScript libraries within the libraries/js directory and informed in the manifest.json are loaded and included in the Client header, starting to apply with part of the Client JavaScript code, as each one is loaded. For JS libraries to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the PHP extension. The content of the file needs to obey a syntax and XML structure. All JS code needs to be inside a <script> root tag. The order in which each file is informed in manifests.json is completely respected in the loading process as well. Stage 7 At this stage, all Pieces inside the pieces folder and informed in manifest.json are loaded. The JavaScript and CSS code of each Piece is loaded to the client and are active on the Client, while the JSON and HTML code of each Piece is stored in cache to be used whenever each component is rendered in Pages that use them. For Pieces to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the HTM extension. The name of the Piece file can only contain letters. The content of the file needs to obey a syntax and XML structure. All Piece code needs to be inside a <piece> root tag. The for attribute of the piece root tag needs to have the value lightmin-engine. The name attribute of the piece root tag must have the value equal to the name of the Piece file, however, without the extension. The tag style (containing CSS), variables (containing JSON), code (containing HTML) and script (containing JavaScript) need to be present in the Piece file. The CSS code of Piece cannot use ID Selectors. The CSS code of Piece cannot use Tag Selectors. The CSS code of Piece cannot use Type Selectors. The CSS code of Piece cannot use Keyframes. The Class Selectors of the Piece CSS code must have a name following the format ".Piece_PieceFileNameWithoutExtension_TheClassName {}". If the CSS code of Piece has a z-index attribute, it cannot be greater than 922000. The JSON code of Piece needs to have a valid syntax. No one Piece HTML code tags can use the style attribute for inline styles. No one Piece HTML code tags can use the id attribute. The HTML code of Piece cannot make use of the html, head, body, style and script tags. All JavaScript code of Piece must be within a class. There can only be one class in the JavaScript code of Piece. The class name involving all the JavaScript code of Piece must have a name that follows the format "class Piece_PieceFileNameWithoutExtension {}". The CSS code and JSON code of the Piece cannot contain the characters < and > outside of comments. Stage 8 At this stage, all Screens inside the sceens folder and informed in manifest.json are loaded. The HTML code of each Screen is loaded to the Client, but they are disabled until the Screen is enabled using the Lightmin Engine JavaScript API. Anyway, the HTML code of each screen will already be part of the Client. For Screens to be considered valid, they need to respect the rules below, and if they do not respect any of the rules, they are ignored in the loading process and are not included in the Client. The file needs to have the HTM extension. The name of the Piece file can only contain letters. The content of the file needs to obey a syntax and XML structure. All Screen code needs to be inside a <screen> root tag. The for attribute of the screen root tag needs to have the value lightmin-engine. The name attribute of the screen root tag must have the value equal to the name of the Screen file, however, without the extension. No one Screen HTML code tags can use the style attribute for inline styles. The HTML code of Screen cannot make use of the html, head, body, style and script tags. Stage 9 At this stage, Engine checks the code present in client.php you wrote. He will ensure that the rules below are respected... No HTML code tags in client.php can use the style attribute for inline styles. The script tag cannot be used within client.php. The style tag cannot be used within client.php. At this stage, Engine also does some additional jobs, such as preparing all window tags present within Client for use, and other things. Stage 10 At the final stage, Engine only finishes loading work, removes the loading screen, run callbacks to warn that the load has been completed and other things. Client: Building the Client layout Building your client.php is very simple. You just need to write your Client HTML code inside the client.php file. Your Website client.php file should look like the image below, however, you should not edit ANYTHING in the red part, only in the green part. In the image below, the green part indicates where exactly you should insert your Client HTML code. Note that you don't need to insert the html, head or body tags, you just need to write the HTML code exactly as you would write it inside the body tags of a regular HTML document. While you write your Client HTML code, you need to keep in mind the idea of setting aside a space in your Client, where you will place a Window that will be the place where the Client content will update as the user navigates through your Website. Where to put your CSS and JS code The Engine does not allow placing CSS and JavaScript code in client.php for some reasons, such as greater organization, greater optimization, among others. Therefore, all CSS and JavaScript code of your website (except Pieces CSS/JS code) must go to the libraries folder. You can read this topic for more details on exactly how to do this. Medias - Where to place Website Media files On websites created using Lightmin Engine, all media files such as Images, Videos, Icons and others must go to the medias folder. This is to ensure greater organization on Lightmin-based websites. Image files should go to medias/images, icon files should go to medias/icons, sound or music files should go to medias/sounds, plain text files should go to medias/others.

This organization will also allow you to be able to find resources (like images) easily using Lightmin Engine JavaScript API, whether in your Pages, Pieces or whatever you want! Libaries - Where to place Website CSS & JS code Any website worth its salt needs CSS and JavaScript codes to function. With websites made in Lightmin, it's the same thing. Although Lightmin provides a platform for you to create your own dynamic websites, you still need to program the JavaScript that will make your website come to life and the CSS that will make your website look however you want.

This is a rule that I strongly recommend you follow while developing websites with Lightmin: All CSS and JavaScript code of your ENTIRE website MUST be inside your website libraries folder. Following this rule will allow for MUCH greater organization of your Website, not to mention that this will facilitate future maintenance that is carried out on your Lightmin website.

JavaScript files should look like this, and must have the PHP extension... <?php /* This is a example JavaScript file for the Lightmin website */ ?> <script> function exampleFunction() { } </script> The events window.onload and document.onload will not work for your JavaScript libraries because they are loaded at runtime on the loading screen, so if you need to run some code right after your library script is loaded by the Engine, just include a method called LE_OnLoad_YourJsLibFileName() in your script. So, if you have a JavaScript library called MyJsLibrary.php, just include the function LE_OnLoad_MyJsLibrary() method in your script and it will be executed as soon as your script is loaded by the Engine. If you want to run code once the Engine has COMPLETELY finished initial loading the Website, you can add function LE_OnLoadEngineFinished() to one of your library JavaScript files. But add this function ONLY ONCE throughout your website. Only ONE of this function can exist PER Website, to avoid conflicts when receiving the callback. CSS files should look like this, and must have the PHP extension too... <?php /* This is a example CSS file for the Lightmin website */ ?> <style> .exampleCssClass { } </style> These files must have the PHP extension as this will allow code editors (like VS Code) to provide auto-completion while you write your CSS and JavaScript code, as well as allowing PHP code to be merged with your CSS and JavaScript code, if you so desire.

Either way, every file that contains CSS should go in directory libraries/css on your Website, while every file that contains JavaScript code should go in directory libraries/js on your Website. And of course, both type of files must comply with the examples mentioned above, otherwise the Engine will not be able to compile them at the time of initial loading, which could result in problems running your Website. Manifest File As you already know, websites created with Lightmin Engine can have multiple Pieces, Screens and Libraries. All of these items help to create your website and make your work easier when developing it. However, upon initial loading of your website, the Engine loads all these items and then compiles them in the Client, at runtime. Once compiled, they will be available for use across your website. However, for the Engine to know that these items exist, you need to inform it. And that's where the manifest.json file comes in.

The manifest.json file is a JSON file and contains some arrays in which you must enter the path to the Libraries (CSS and JavaScript), Pieces and Screens files. A manifest.json file should look like this... { "baseTitle": "Example Website", "defaultPage": "pages/home.php", "cssLibraries": [ "libraries/css/example.php" ], "javascriptLibraries": [ "libraries/js/example.php" ], "websitePieces": [ "pieces/TextField.htm", "pieces/ImageDisplay.htm" ], "websiteScreens": [ "screens/Preferences.htm" ] } Now, let's see what each variable (array) in manifest.json means. cssLibraries: In this variable you must enter the path to each CSS file within the libraries folder on your Website. javascriptLibraries: In this variable you must enter the path to each JavaScript file within the libraries folder on your Website. websitePieces: In this variable you must enter the path to each Piece file within the pieces folder on your Website. websiteScreens: In this variable you must enter the path to each Screens file within the screens folder on your Website. Remember that you must provide the paths to the files, taking into account that the root folder is the same folder where file client.php is located. Be careful to not save file manifest.json containing syntax errors. If the file does not comply with the correct JSON syntax, the Lightmin Engine will not be able to load it correctly and your Website may be interrupted. Also, never remove any existing variables (array) from file manifest.json, even if the array is empty. Settings File The Engine, in its background, performs a lot of functions to abstract as much as possible for you, so that you don't need to worry about the foundation or code base for building your Dynamic Website. However, several things regarding the Engine operation can be configured, such as the background colors of the loading screen, the Website icon, the Client language and MANY other things.

The settings.php file allows you to easily configure settings. The Engine has several variables that have values with default settings. If you define an existing variable in settings.php by defining a new value for the variable, you will be defining a new configuration as desired. You can use the PHP code below to set new values for variables. $settings->SetVariablePrimitiveValue("variableName", "The New Value"); If you want to set the default Engine value for the variable, just delete the line where you are setting a value for it and you're done! That variable will now use the Engine default value! This also means that if you don't define any variables in the settings.php file, then all existing variables will be using Engine default values!

Lightmin JS API: About Lightmin Engine JavaScript API is accessible from ANY part of your website. Whether in client.php, in libraries/js, in Pieces or even in Pages. Lightmin JavaScript APIs provide several useful functions as well as control functions for your website to facilitate the development of your website and speed up the process where possible. Lightmin JS API: Settings.Get() This method allows you to get the value of ANY settings variable defined on your website through settings.php and also allows you to get the value of settings variables that are not defined in settings.php, getting the default value of Engine for that variable. //- Settings.Get(settingName); // STRING settingName - The name of settings variable that you want to get the value. var value = Settings.Get("websiteLang"); Lightmin JS API: Settings.GetManifestBaseTitle() This method allows you to obtain the value defined in the baseTitle variable in your Website manifest.json file. //- Settings.GetManifestBaseTitle(); var baseTitle = Settings.GetManifestBaseTitle(); Lightmin JS API: Settings.GetManifestDefaultPage() This method allows you to obtain the value defined in the defaultPage variable in your Website manifest.json file. //- Settings.GetManifestDefaultPage(); var defaultPage = Settings.GetManifestDefaultPage(); Lightmin JS API: Common.SendLog() This method allows you to send logs (message only) to the browser console, in the context of the Engine. //- Common.SendLog(logType, message); // STRING logType - The type of log that will be sent. "E" for errors, "W" for warnings, and "L" for simple logs. // STRING message - The log message. Common.SendLog("L", "Hello World!"); Lightmin JS API: Utils.RemoveAllListeners() This method removes all Listeners registered in a DOM element. Note that if you have a reference to an element in some variable, and then you use this method on the element, the variable will lose reference to the element on which you used this method. So if you were storing a reference to that element in a variable, you will need to reference it again after using this method on it. //- Utils.RemoveAllListeners(domElement); // ELEMENT domElement - The element that will have all listeners removed. Utils.RemoveAllListeners(document.getElementById("someElementId")); Lightmin JS API: Utils.GetCurrentFaviconSrc() This method returns the HREF attribute value of the current Favicon. It is useful for checking which Favicon is currently displayed on the Website, and can be called frequently without problems, as it has a Cache system for future checks. //- Utils.GetCurrentFaviconSrc(); var currentFaviconSrc = Utils.GetCurrentFaviconSrc(); Lightmin JS API: Utils.ChangeFavicon() This method allows you to dynamically change the Website Favicon. This can be called frequently without problems, as it has a Cache system for future checks. //- Utils.ChangeFavicon(newFaviconSrc); // STRING newFaviconSrc - The new HREF of the icon. It can be a URI or URL. Utils.ChangeFavicon("icons/myFavicon.ico"); Lightmin JS API: HttpRequest The Lightmin Engine HttpRequest class is nothing more than a port to Lightmin, of the "Frontend Http Request" JS library created by Marcos Tomaz. Frontend HTTP Request is a library that aims to facilitate HTTP requests made with pure Javascript in the Frontend of your Websites. It was developed to make working with AJAX easier.

What this library does is abstract most of the code needed to make HTTP requests, reducing the amount of code needed and the repetitiveness you have when using "XmlHttpRequest" objects from Javascript. With this library you can make HTTP requests like POST, GET or File Upload in an easy and fast way, with little code and with easy to use Callbacks so that you can easily change the UI of your Website while the Request is being made.

Lightmin Engine HttpRequest class can be used easily and you can refer to the Documentation (ReadMe) of "Frontend Http Request" to learn how to use this class easily. Just read the Documentation by clicking on this link and you will know how to use the HttpRequest class easily! Lightmin JS API: InputValidator The Lightmin Engine InputValidator class is nothing more than a port to Lightmin, of the "Frontend Input Validator" JS library created by Marcos Tomaz. The Frontend Input Validator is a Javascript library to be used in the Frontend of your website. The objective of this library is to provide a simple, easy to use and low code system for validating text or file inputs in your website Frontend interface. Which is very common in all types of websites, such as registration forms for example.

Lightmin Engine InputValidator class can be used easily and you can refer to the Documentation (ReadMe) of "Frontend Input Validator" to learn how to use this class easily. Just read the Documentation by clicking on this link and you will know how to use the InputValidator class easily! Lightmin PHP API: About Lightmin Engine's PHP API allows you to do some useful things for your website, however, on the PHP side. The downside is that so far, Lightmin Engine's PHP APIs will only work in your client.php file. Note that PHP APIs will not work in your JavaScript codes and vice versa... Lightmin PHP API: Common::GetManifestBaseTitle() This method returns the base title of your Website, which is configured in the "manifest.json" file. //- Common::GetManifestBaseTitle(); $websiteBaseTitle = Common::GetManifestBaseTitle(); Lightmin PHP API: Common::GetManifestDefaultPage() This method returns the default page of your Website, which is configured in the "manifest.json" file. //- Common::GetManifestDefaultPage(); $websiteDefaultPage = Common::GetManifestDefaultPage();
