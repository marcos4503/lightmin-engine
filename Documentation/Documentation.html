What is Lightmin Engine Lightmin Engine (a combination of the words Light and Minimalistic) is an Engine aimed for creating Websites. It aims to provide a simple, fast, easy and powerful way to create dynamic websites, that is, websites that behave like applications! Dynamic websites are responsive websites, which often behave like applications, having a loading screen when the user accesses that website, and then the user no longer leaves that same page and only its content, as well as the interface, changes as the user navigates and uses the website, exactly how an application behaves.

The Lightmin Engine aims to be an alternative for people who do not want or cannot use much more complex Frameworks/Engines that require a greater learning curve, such as Flutter, React, Angular, among others.

The Engine is written in pure PHP, HTML, CSS and JavaScript! This means that to use this Engine and develop your website, all you will need is a server that runs PHP and a program to edit HTML, CSS, JavaScript and PHP codes! You won't need anything more than that and you'll be able to create your dynamic website in a matter of minutes after understanding the basics of this Engine!

Lightmin Engine Logo What is the dynamic behind Lightmin? Lightmin is an engine for creating dynamic websites based on Pieces, Pages and a Client. These are the three main pillars of Lightmin. The Client behaves like the base design of your dynamic website, being the base interface for your dynamic website, like mobile apps. Once the user enters your dynamic website, they will be entering the Client and will not leave there, unless they access another website or close the browser tab. The Pages work like standard HTML pages, however, they are rendered INSIDE the Client and whenever the user clicks on a link or button, and it is necessary to change to a new page, the user will not leave the Client, but instead, the current Page inside the Client will be unloaded and then the next Page will be loaded and then displayed inside the Client! You can watch the video below to understand better. In the example video below, I will show how MEGA dynamic website works. It wasn't made using Lightmin, but it works exactly like websites made with Lightmin! Using the MEGA website as an example... The Pieces act like Components in applications. Basically, Pieces are REUSABLE Components in which you use HTML, JavaScript and CSS to create them. Once you create a Piece, such as example, a Smart Text Field, you can render that Piece as many times as you want within Pages! And yes! If you decide to edit your Piece in the future, you will only need to edit the Piece file and then all existing Pieces on any Page will be updated too! The concept of Pieces makes it easier to maintain, organize and develop your websites!

Add the concepts of Client plus Pages plus Pieces plus HTML, CSS, PHP and the JavaScript that you already know and are used to, and then we have the Lightmin Engine! What do I need before start? The Lightmin Engine was developed to enable developers to create their dynamic websites (or web applications) in the most enjoyable, quick and easy way possible. It was developed so that you don't need to have compiler programs or anything similar on your PC, all you need is your standard code editor. But there are some requirements necessary for you to use Lightmin, some are optional. See below. Prior knowledge of HTML, CSS, JavaScript, JSON, PHP and how to create/host Websites. A server that runs PHP. Everyone who will access your website will need Browsers that support JavaScript. An HTML, JavaScript, CSS and PHP code editor. File structure of a Lightmin Website When downloading the Lightmin base Website from the Github repository, before editing it and making your own Website from it, you will see several files and folders. In the table below you can see what each folder and file is, as well as explanations about each one. 20%=>20%=>auto File=>Type=>Description engine=>Folder=>These are the Lightmin Engine core files, they are the files that are behind everything, making everything work. You do not need (and do not recommend) touching any of the files inside this folder. However, if in the future you want to update the Lightmin that is running behind your Website, you will only need to replace this folder with the "engine" folder of the new version of Lightmin Engine! libraries=>Folder=>Inside this folder you must place your Website CSS and JS files. I STRONGLY recommend that ALL of your Website CSS be inside this folder (except the "Pieces" CSS) so that organization is maintained, and to avoid clutter. medias=>Folder=>Inside this folder you must place ALL the media files from your Website. Be it a pure text file, image or even video. Files present here can also be easily accessed via JavaScript using the Lightmin Engine JS API! pages=>Folder=>In this folder you must place ALL the "Pages" files from your Lightmin Website. Any "Page" file that you intend to render within the Client must be present here. The "Page" files can contain JavaScript within them, without any problems, but it is recommended that they only be small scripts. Any large script should go to the "libraries" folder for greater organization of the Website. pieces=>Folder=>In this folder you must place all the HTM files of the "Pieces" that you created for your Website. When downloading the base website from the Github repository, some standard "Pieces" will come so that you have at least the basics to get started! The "Pieces" must contain their JavaScript code within the HTM file, regardless of the size of the script, along with the HTML and CSS code of the "Piece", for greater organization. Additionally, this makes it easier to share "Pieces" between different Lightmin websites. screens=>Folder=>In this folder, you must place all the "Screen" files for your Website. These files works as additional screens that cover your Website Client when called by JavaScript code, and this way you can use them to extend the base interface beyond just the Client interface, creating additional screens with different layouts and interfaces to perform smaller tasks on your Website, such as Login screens, Preferences screens, etc. client.php=>File=>This is your Website "Client" file, this is your Website central file. This is where you will program the layout and base interface of your Website, and it is where you will display "Pages" and other things. manifest.json=>File=>This is the Manifest of your Lightmin Website. Here in this file you must declare all the "Pieces", "Screens" and "Libraries (CSS and JS)" files present on your Lightmin Website. Only then will these files work and be compiled at runtime on the Client. settings.php=>File=>In this file you can configure any type of configurable thing for your Lightmin Website, which can range from colors of Engine elements, to transition animations between "Pages" in the Client. Medias - Where to place Website Media files On websites created using Lightmin Engine, all media files such as Images, Videos, Icons and others must go to the medias folder. This is to ensure greater organization on Lightmin-based websites. Image files should go to medias/images, icon files should go to medias/icons, sound or music files should go to medias/sounds, plain text files should go to medias/others.

This organization will also allow you to be able to find resources (like images) easily using Lightmin Engine JavaScript API, whether in your Pages, Pieces or whatever you want! Libaries - Where to place Website CSS & JS code Any website worth its salt needs CSS and JavaScript codes to function. With websites made in Lightmin, it's the same thing. Although Lightmin provides a platform for you to create your own dynamic websites, you still need to program the JavaScript that will make your website come to life and the CSS that will make your website look however you want.

This is a rule that I strongly recommend you follow while developing websites with Lightmin: All CSS and JavaScript code of your ENTIRE website MUST be inside your website libraries folder. Following this rule will allow for MUCH greater organization of your Website, not to mention that this will facilitate future maintenance that is carried out on your Lightmin website.

JavaScript files should look like this, and must have the PHP extension... <?php /* This is a example JavaScript file for the Lightmin website */ ?> <script> function exampleFunction() { } </script> The events window.onload and document.onload will not work for your JavaScript libraries because they are loaded at runtime on the loading screen, so if you need to run some code right after your library script is loaded by the Engine, just include a method called LE_OnLoad_YourJsLibFileName() in your script. So, if you have a JavaScript library called MyJsLibrary.php, just include the function LE_OnLoad_MyJsLibrary() method in your script and it will be executed as soon as your script is loaded by the Engine. If you want to run code once the Engine has COMPLETELY finished initial loading the Website, you can add function LE_OnLoadEngineFinished() to one of your library JavaScript files. But add this function ONLY ONCE throughout your website. Only ONE of this function can exist PER Website, to avoid conflicts when receiving the callback. CSS files should look like this, and must have the PHP extension too... <?php /* This is a example CSS file for the Lightmin website */ ?> <style> .exampleCssClass { } </style> These files must have the PHP extension as this will allow code editors (like VS Code) to provide auto-completion while you write your CSS and JavaScript code, as well as allowing PHP code to be merged with your CSS and JavaScript code, if you so desire.

Either way, every file that contains CSS should go in directory libraries/css on your Website, while every file that contains JavaScript code should go in directory libraries/js on your Website. And of course, both type of files must comply with the examples mentioned above, otherwise the Engine will not be able to compile them at the time of initial loading, which could result in problems running your Website. Manifest File As you already know, websites created with Lightmin Engine can have multiple Pieces, Screens and Libraries. All of these items help to create your website and make your work easier when developing it. However, upon initial loading of your website, the Engine loads all these items and then compiles them in the Client, at runtime. Once compiled, they will be available for use across your website. However, for the Engine to know that these items exist, you need to inform it. And that's where the manifest.json file comes in.

The manifest.json file is a JSON file and contains some arrays in which you must enter the path to the Libraries (CSS and JavaScript), Pieces and Screens files. A manifest.json file should look like this... { "cssLibraries": [ "libraries/css/example.php" ], "javascriptLibraries": [ "libraries/js/example.php" ], "websitePieces": [ "pieces/TextField.htm", "pieces/ImageDisplay.htm" ], "websiteScreens": [ "screens/Preferences.htm" ] } Now, let's see what each variable (array) in manifest.json means. cssLibraries: In this variable you must enter the path to each CSS file within the libraries folder on your Website. javascriptLibraries: In this variable you must enter the path to each JavaScript file within the libraries folder on your Website. websitePieces: In this variable you must enter the path to each Piece file within the pieces folder on your Website. websiteScreens: In this variable you must enter the path to each Screens file within the screens folder on your Website. Remember that you must provide the paths to the files, taking into account that the root folder is the same folder where file client.php is located. Be careful to not save file manifest.json containing syntax errors. If the file does not comply with the correct JSON syntax, the Lightmin Engine will not be able to load it correctly and your Website may be interrupted. Also, never remove any existing variables (array) from file manifest.json, even if the array is empty. Settings File The Engine, in its background, performs a lot of functions to abstract as much as possible for you, so that you don't need to worry about the foundation or code base for building your Dynamic Website. However, several things regarding the Engine operation can be configured, such as the background colors of the loading screen, the Website icon, the Client language and MANY other things.

The settings.php file allows you to easily configure settings. The Engine has several variables that have values with default settings. If you define an existing variable in settings.php by defining a new value for the variable, you will be defining a new configuration as desired. You can use the PHP code below to set new values for variables. $settings->SetVariablePrimitiveValue("variableName", "The New Value"); If you want to set the default Engine value for the variable, just delete the line where you are setting a value for it and you're done! That variable will now use the Engine default value! This also means that if you don't define any variables in the settings.php file, then all existing variables will be using Engine default values!

In the table below, you can see all the variables that exist in the Engine, which you can configure to define any value you want, as long as the value you define respects the value type of that variable. For example, you cannot define a value of type INT for a variable of type STRING. 44%=>6%=>auto Variable=>Type=>Description websiteBaseTitle=>STRING=>Sets the base title for your Website. This title will be displayed in the Browser tab and will come before the title of the Page currently loaded and being displayed. Generally you will want to set the main title of the website itself. For example, if the website is for Honda, you will want to set the value "Honda" for this variable. websiteFaviconNoNotificationsUri=>STRING=>Sets the image (in ICO format) to be used as the website Favicon while there are no Notifications being displayed using the Lightmin Engine Notifications API. You must provide the path, taking into account that the root is the folder where "client.php" is located. websiteFaviconWithNotificationsUri=>STRING=>Sets the image (in ICO format) to be used as the website Favicon while THERE ARE notifications being displayed using the Lightmin Engine Notifications API. You must provide the path, taking into account that the root is the folder where "client.php" is located. websiteLang=>STRING=>Defines the language of your Client page. It must be defined in accordance with the ISO-8859-1 format. For example "pt-BR" or "en-US". websiteCharSet=>STRING=>Defines the charset to be used by your website Client. The default is "UTF-8". websitePrimaryFontFamily=>STRING=>Defines the primary Font Family for your website text. websiteSecondaryFontFamily=>STRING=>Defines the secondary Font Family for your website text. websiteFontSizePx=>INT=>Sets the default Font size for your website. browserColor=>STRING=>Defines the color that your Website will have in the Browser Tab. Generally this will only work for Google Chrome on mobile devices. Must be set in Hexadecimal color format. loadScreenLogoUri=>STRING=>Defines the image that will be used as your website loading logo when the user enters. You must provide the path, taking into account that the root is the folder where "client.php" is located. loadScreenMessage=>STRING=>Defines the message that will appear below the logo on the loading screen. loadScreenBackgroundColorHexGradientStart=>STRING=>Sets the starting color of the gradient on the background of your website initial loading screen. Must be set in Hexadecimal color format. loadScreenBackgroundColorHexGradientEnd=>STRING=>Sets the ending color of the gradient on the background of your website initial loading screen. Must be set in Hexadecimal color format. loadScreenStyledBackgroundColorHex=>STRING=>Sets the background color that will be present behind the initial loading screen, as a drop shadow in the exit transition. Must be set in Hexadecimal color format. loadScreenForegroundColorHex=>STRING=>Sets the filling color of the loading bar on the Engine initial loading screen. Must be set in Hexadecimal color format. loadScreenBackgroundColorHex=>STRING=>Sets the background color of the loading bar on the Engine initial loading screen. Note: If you try to define a variable that does not exist in the Engine, in the settings.php file, it will generate runtime errors on your Website when it tries to read the settings. This can cause problems and even stop your website from running, resulting in errors. Therefore, only define existing variables, that is, variables that are listed in the table above. Be careful when saving file settings.php not to save the file with PHP syntax errors or similar things as well. If there is any error of this type in the settings.php file, the Engine will not be able to load the settings and the execution of your website will be affected. Lightmin JS API: Settings.Get() This method allows you to get the value of ANY settings variable defined on your website through settings.php and also allows you to get the value of settings variables that are not defined in settings.php, getting the default value of Engine for that variable. //- Settings.Get(settingName); // STRING settingName - The name of settings variable that you want to get the value. var value = Settings.Get("websiteBaseTitle"); Lightmin JS API: Common.SendLog() This method allows you to send logs (message only) to the browser console, in the context of the Engine. //- Common.SendLog(logType, message); // STRING logType - The type of log that will be sent. "E" for errors, "W" for warnings, and "L" for simple logs. // STRING message - The log message. Common.SendLog("L", "Hello World!"); Lightmin JS API: Utils.RemoveAllListeners() This method removes all Listeners registered in a DOM element. Note that if you have a reference to an element in some variable, and then you use this method on the element, the variable will lose reference to the element on which you used this method. So if you were storing a reference to that element in a variable, you will need to reference it again after using this method on it. //- Utils.RemoveAllListeners(domElement); // ELEMENT domElement - The element that will have all listeners removed. Utils.RemoveAllListeners(document.getElementById("someElementId")); Lightmin JS API: Utils.GetCurrentFaviconSrc() This method returns the HREF attribute value of the current Favicon. It is useful for checking which Favicon is currently displayed on the Website, and can be called frequently without problems, as it has a Cache system for future checks. //- Utils.GetCurrentFaviconSrc(); var currentFaviconSrc = Utils.GetCurrentFaviconSrc(); Lightmin JS API: Utils.ChangeFavicon() This method allows you to dynamically change the Website Favicon. This can be called frequently without problems, as it has a Cache system for future checks. //- Utils.ChangeFavicon(newFaviconSrc); // STRING newFaviconSrc - The new HREF of the icon. It can be a URI or URL. Utils.ChangeFavicon("icons/myFavicon.ico"); Lightmin JS API: HttpRequest The Lightmin Engine HttpRequest class is nothing more than a port to Lightmin, of the "Frontend Http Request" JS library created by Marcos Tomaz. Frontend HTTP Request is a library that aims to facilitate HTTP requests made with pure Javascript in the Frontend of your Websites. It was developed to make working with AJAX easier.

What this library does is abstract most of the code needed to make HTTP requests, reducing the amount of code needed and the repetitiveness you have when using "XmlHttpRequest" objects from Javascript. With this library you can make HTTP requests like POST, GET or File Upload in an easy and fast way, with little code and with easy to use Callbacks so that you can easily change the UI of your Website while the Request is being made.

Lightmin Engine HttpRequest class can be used easily and you can refer to the Documentation (ReadMe) of "Frontend Http Request" to learn how to use this class easily. Just read the Documentation by clicking on this link and you will know how to use the HttpRequest class easily! Lightmin JS API: InputValidator The Lightmin Engine InputValidator class is nothing more than a port to Lightmin, of the "Frontend Input Validator" JS library created by Marcos Tomaz. The Frontend Input Validator is a Javascript library to be used in the Frontend of your website. The objective of this library is to provide a simple, easy to use and low code system for validating text or file inputs in your website Frontend interface. Which is very common in all types of websites, such as registration forms for example.

Lightmin Engine InputValidator class can be used easily and you can refer to the Documentation (ReadMe) of "Frontend Input Validator" to learn how to use this class easily. Just read the Documentation by clicking on this link and you will know how to use the InputValidator class easily!
